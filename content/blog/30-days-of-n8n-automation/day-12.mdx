## The Architectural Mandate: Why Wait Nodes Fail Without Context

When amateurs attempt to build an automated email follow up sequence in n8n, they build a linear chain:
Send Email 1 ➡️ Wait 3 Days ➡️ Send Email 2.

This is a catastrophic architectural error. It is the fastest way to ruin your domain reputation and infuriate a prospect.

Imagine a VP of Engineering replies to your first email on Day 2, saying, "Yes, let's book a call for tomorrow." Because the amateur workflow is linear, it does not know the prospect replied. On Day 3, it blindly fires the second email: "Hey, just bubbling this up to the top of your inbox!" You instantly look like a spam bot, and the deal dies.

Professional RevOps architecture requires the **Check-Wait-Check pattern**. Before a server ever sends a follow-up payload, it must query the master database to verify the current "State" of the lead. If the state has changed to REPLIED, the automation must terminate immediately.

## Infrastructure Prerequisites (The Hard Requirements)

To build a stateful sequence, you cannot use a standard Gmail or Outlook node. Native email clients do not easily expose robust webhooks for reply tracking. You must use a dedicated transactional email engine.

### 1. The Core Execution Server

Because this workflow relies on pausing executions for days at a time, your n8n instance must have stable database persistence. If your server reboots and memory is wiped, your paused follow-ups will fail. You must execute this on a production-grade n8n Cloud instance or a highly stable [Vultr VPS deployment](https://www.vultr.com/?ref=9859101-9J).

### 2. The Transactional Email Engine (Mandatory)

To execute Phase 2 of this n8n workflow, you must generate a transactional v3 API key from a dedicated email engine that supports inbound parsing.

If you do not have one configured, pause this tutorial and deploy your free Brevo infrastructure.

Brevo (formerly Sendinblue) is the required infrastructure for this blueprint because it provides native API nodes within n8n, allows for deep contact attribute tagging, and handles all CAN-SPAM/GDPR unsubscribe compliance automatically. You cannot proceed to the routing logic without this API key.

## Phase 1: Database Preparation and The Reply Catcher

Before we build the outbound sequence, we must build the system that catches the inbound replies. We need a way to tell n8n that a human has responded.

### Step 1: Architecting the Brevo Attributes

Log into your Brevo dashboard. Navigate to Contacts ➡️ Settings ➡️ Contact Attributes.

Create a new Text attribute named `LEAD_STAGE`.

The default state for any new lead entering your system will be `COLD`.

### Step 2: The Independent Webhook (The Reply Catcher)

You must create a separate, micro-workflow in n8n purely to listen for replies.

Create a new n8n workflow named **Microservice: Brevo Reply Listener**.

Add a Webhook node.

In Brevo's settings, navigate to Webhooks and point "Inbound Email" or "Reply" events to this n8n Webhook URL.

When a prospect replies to your cold email, Brevo will instantly ping this Webhook with the prospect's email address.

Attach a Brevo node to the Webhook:
- **Resource**: Contact
- **Operation**: Update
- **Email**: Map the email from the webhook payload.
- **Update Attributes**: Set `LEAD_STAGE` to `REPLIED`.

This microservice runs invisibly in the background 24/7. It guarantees that the master database is always perfectly synced with reality.

## Phase 2: Building the Outbound Sequence (The Entry Point)

Now, we build the actual automated drip campaign. Create a new n8n workflow named **Core: Autonomous Outbound Sequence**.

![n8n workflow diagram for automated email follow-ups with Brevo](/images/blog/automated-email-follow-up-n8n-brevo-unnamed-11.webp)

### Node 1: The Trigger

You can trigger this sequence in multiple ways:
- Connecting it directly to the end of the Lead Capture engine we built in [Day 8](/blog/capture-n8n-lead-data-from-wordpress-elementor).
- Triggering it when a lead is dragged into a specific column in your CRM.
- Using a Brevo Trigger node listening for "Contact Added to List" (e.g., your "Q2 Cold Outreach" list).

### Node 2: The Initial Send

We do not write email copy inside n8n. Hardcoding text inside an automation tool makes it impossible for your marketing team to A/B test subject lines. You must build your templates inside Brevo and use n8n merely to trigger them.

Attach a Brevo node to your trigger:
- **Resource**: Transactional Emails
- **Operation**: Send
- **Recipient Email**: `{{ $json.email }}`
- **Template ID**: Enter the integer ID of your first outreach template (e.g., 12).

*Architect's Note*: Ensure this template utilizes the Enrichment data we gathered in Day 11. If you enriched their company size, use Brevo's template variables to say: "I noticed `{{ contact.ENRICHED_COMPANY }}` recently crossed `{{ contact.ENRICHED_EMPLOYEES }}` employees..." Hyper-personalization is how you avoid the spam folder.

## Phase 3: The Stateful Suspension (The Wait Node)

At this exact moment, the lead has received your first email. Now the system must suspend itself to simulate natural human patience.

Add a Wait node to the canvas:
- **Resume**: After time interval
- **Amount**: 3
- **Unit**: Days

*How n8n Handles Memory*: When an execution hits a Wait node that is longer than a few minutes, n8n does not keep the process actively running (which would crash your server CPU). It takes the entire payload, writes it securely to its internal SQLite or Postgres database, and terminates the active process. Exactly 72 hours later, a cron job wakes the execution up, pulls the data from the database, and resumes the flow exactly where it left off.

This is why a stable hosting environment is a hard prerequisite.

## Phase 4: The Intelligence Query

72 hours have passed. The execution has woken up. Before we send Email 2, we must execute the "Check" phase of the Check-Wait-Check architecture.

We must ask the Brevo database: "Did they reply while I was asleep?"

Attach a Brevo node directly after the Wait node:
- **Resource**: Contact
- **Operation**: Get
- **Identifier**: Email
- **Value**: Map the email address from the original trigger node (e.g., `{{ $('Trigger').item.json.email }}`).

This node pulls the absolute freshest data available for that prospect.

## Phase 5: The Logic Gate (IF Node)

We now have the fresh data. We must evaluate it using the core concepts established in [Day 6](/blog/essential-n8n-core-nodes-by-alfaz-mahmud-rizve).

Attach an IF Node to the Get Contact node:
- **Condition**: String
- **Value 1**: `{{ $json.attributes.LEAD_STAGE }}`
- **Operation**: Not Equal
- **Value 2**: `REPLIED`

### The Routing Logic:

**The True Branch (Continue Pestering)**: If `LEAD_STAGE` is NOT equal to `REPLIED` (meaning it is still `COLD`), the prospect ignored your first email. The workflow routes down the True branch. You attach another Brevo Send node here, mapped to Template ID 13 (Your polite follow-up).

**The False Branch (Terminate)**: If `LEAD_STAGE` IS equal to `REPLIED`, the prospect responded. The Webhook microservice we built in Phase 1 did its job. The workflow routes down the False branch. You attach a NoOp node here (or simply leave it blank) to silently terminate the sequence. You have successfully prevented an embarrassing automated email.

You can chain this exact logic `Wait ➡️ Get Contact ➡️ IF ➡️ Send` as many times as you need to build a 5-step or 7-step sequence. I utilize Exponential Backoff (Day 3, Day 7, Day 14, Day 30) to stay top-of-mind without triggering spam filters.

## Advanced Engineering: The Business Hours Filter

A clear indicator that you are using an automated email follow up tool is when a prospect receives a highly technical B2B pitch at 4:00 AM on a Sunday.

If you set a Wait node for "3 Days", and the original email was sent on a Thursday, the follow-up will trigger on Sunday. We must engineer a JavaScript filter to respect standard business hours.

![Business hours filter logic](/images/blog/automated-email-follow-up-n8n-brevo-unnamed-13.webp)

Before the "Send Email 2" node, insert a Code Node with this logic:

```javascript
// Check current Server Time
const now = new Date();
const dayOfWeek = now.getDay(); // 0 = Sunday, 1 = Monday, etc.
const currentHour = now.getHours();

let delayNeeded = false;
let hoursToWait = 0;

// If it's Saturday (6)
if (dayOfWeek === 6) {
    delayNeeded = true;
    hoursToWait = 48; // Push to Monday
} 
// If it's Sunday (0)
else if (dayOfWeek === 0) {
    delayNeeded = true;
    hoursToWait = 24; // Push to Monday
}
// If it's a weekday but before 8 AM
else if (currentHour < 8) {
    delayNeeded = true;
    hoursToWait = 8 - currentHour;
}
// If it's a weekday but after 5 PM (17:00)
else if (currentHour >= 17) {
    delayNeeded = true;
    hoursToWait = 24 - currentHour + 8; // Push to 8 AM next morning
}

return {
    delay_needed: delayNeeded,
    wait_hours: hoursToWait
};
```

Following this Code Node, you add one final IF Node. If `delay_needed` is true, route the workflow to a dynamic Wait node that pauses for `wait_hours` before finally executing the Brevo Send node.

This level of rigor separates hobbyists from RevOps architects.
